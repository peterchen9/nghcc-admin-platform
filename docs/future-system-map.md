# 未來系統版圖建議

更新日期：2026-05-29

## 目標

本文件根據 [system-boundary-analysis.md](system-boundary-analysis.md) 的盤點結果，整理哪些功能應保留在 `nghcc-admin-platform`，哪些應拆分成獨立系統，哪些只適合做外部整合入口。

## 建議保留在 admin-platform 的功能

| 功能 | 建議 | 理由 |
| --- | --- | --- |
| 會員 / 關懷 | 保留 | 已有 `modules.eureka`、`members` 資料表、template 與可操作頁面 |
| 詩歌資料庫 | 保留 | 已有完整 app、API、template、media 檔案與上傳流程 |
| 網路影音下載 | 保留為工具 | 已有頁面與 API，但需補安全與外部依賴管理 |
| 使用者管理 | 保留 | 目前是 admin-platform 的基礎管理能力 |
| 選單管理 | 保留 | 控制入口與一般使用者可見功能 |
| 首頁內容編輯 | 保留，但定位為輕量內容管理 | 有 `pages_page` 與 CKEditor，上限應是入口公告/首頁內容，不宜變成完整 CMS |
| QR 報到資料查詢 | 可保留查詢入口 | `checkin_records` 已在 DB，且會員頁會局部使用首次打卡 |

## 應拆分成獨立系統的功能

| 功能 | 建議 | 理由 |
| --- | --- | --- |
| QR 報到系統 | 獨立系統，admin-platform 做查詢入口或管理入口 | 報到通常牽涉掃碼、現場設備、即時流量與活動場景，不宜塞進行政主系統 |
| 折價券系統 | 獨立系統 | 可能牽涉兌換流程、核銷、活動規則與對外使用者入口 |
| 維修系統 / 重修舊好維修系統 | 獨立系統 | 維修案件有自己的狀態流、附件、指派與通知流程 |
| 教會軟硬體維護系統 | 獨立系統或與維修系統合併 | 屬資產/工單管理，生命週期不同於行政資料查詢 |
| 財會 / 奉獻 | 原則上獨立系統 | 涉及敏感金流與權限稽核，不宜只靠一般 admin-platform 權限模型 |

## 外部系統整合點

| 外部系統 | admin-platform 可提供的整合方式 |
| --- | --- |
| QR 報到系統 | 顯示入口連結、查詢會員出席摘要、讀取或同步 `checkin_records` |
| 折價券系統 | 顯示入口連結、依權限導向外部系統，不直接處理核銷 |
| 維修系統 | 顯示入口連結、單一登入或帳號同步、行政首頁顯示待辦摘要 |
| 教會軟硬體維護系統 | 顯示入口連結、資產/維修摘要嵌入 |
| 財會 / 奉獻系統 | 僅做授權入口或摘要儀表板，敏感明細留在專屬系統 |

## 待移除或停用選單

| 選單 | 建議 |
| --- | --- |
| 教育 | 已停用，維持停用直到確認課程教育系統邊界 |
| 奉獻 | 若短期沒有實作，先從一般使用者選單隱藏或標示外部入口 |
| 財會 | 若短期沒有實作，先從一般使用者選單隱藏或標示外部入口 |
| 同工 | 釐清是否為人事、同工名冊或外部系統入口 |
| 交通 | 釐清是否為車輛/接送/交通服事系統 |
| 場地設施 | 釐清是否屬維修/預約/資產管理 |
| 資訊網路 | 釐清是否屬教會軟硬體維護系統 |
| 參考資料 | 若只是文件入口，建議改成 CMS/文件連結；若沒有內容則停用 |
| 聚會活動 | 釐清是否屬 QR 報到、課程活動，或獨立活動系統 |

## admin-platform 定位方案比較

### 方案一：主系統

| 面向 | 優點 | 缺點 |
| --- | --- | --- |
| 功能集中 | 所有行政功能在同一 repo、同一登入、同一資料庫 | 邊界容易失控，奉獻/財會/報到/維修的權限與資料風險會混在一起 |
| 維護 | 部署入口單純 | 任一模組故障可能影響整個行政平台 |
| 資料 | 跨模組查詢容易 | 資料表與 migration 會越來越難控管 |

適合情境：團隊很小、功能都非常輕量、沒有敏感資料或高頻現場操作。

### 方案二：Portal

| 面向 | 優點 | 缺點 |
| --- | --- | --- |
| 功能入口 | admin-platform 負責登入、選單、首頁、會員/詩歌等核心行政查詢 | 需要建立外部系統清冊與入口規則 |
| 邊界 | QR 報到、折價券、維修、財會/奉獻可獨立部署與管理 | 需處理跨系統帳號、權限與連結 |
| 風險 | 敏感或高變動系統不拖累主平台 | 初期文件與治理工作較多 |

適合情境：教會已有多個系統，且需要一個統一入口，但不希望所有功能混成單一大型專案。

### 方案三：CMS

| 面向 | 優點 | 缺點 |
| --- | --- | --- |
| 內容管理 | 適合公告、首頁、參考資料、簡易頁面 | 不適合承載奉獻、財會、QR 報到、維修工單等業務流程 |
| 使用者體驗 | 非工程人員可編輯內容 | 權限、流程、資料模型會被 CMS 結構限制 |
| 現況相容 | 目前已有 `pages_page` 與 CKEditor | 若定位為完整 CMS，會和會員/詩歌/行政功能混淆 |

適合情境：只把 admin-platform 當內容入口與公告平台，而非業務系統。

## 建議定位

建議將 `nghcc-admin-platform` 定位為「Portal + 核心行政查詢平台」，不是完整主系統，也不是純 CMS。

建議邊界：

1. 保留：會員/關懷、詩歌、影音工具、使用者管理、選單管理、首頁內容。
2. Portal 化：外部系統入口、公告、常用連結、簡易摘要。
3. 拆分：QR 報到、折價券、維修、教會軟硬體維護、財會/奉獻。
4. CMS 限縮：只做首頁與簡單內容頁，不承擔完整內容網站。

這個定位最能降低風險：admin-platform 保持穩定入口，敏感或高變動流程各自獨立，日後再透過權限與連結整合。
